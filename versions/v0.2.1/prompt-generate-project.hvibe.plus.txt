LOCK FILE

Before reading the .hvibe.plus file, check if a .hvibe.lock file is present.

If a lock file is present:
- Read it first.
- For every feature marked locked: true, treat its existing implementation as final. Do not regenerate, refactor, or reinterpret it.
- If a locked feature must change to satisfy a new requirement, stop and explicitly flag it to the user before modifying anything. Do not proceed until the user validates the change.
- Focus generation exclusively on features marked locked: false or absent from the lock.

If no lock file is present, proceed normally with full generation.

HVIBE.PLUS READING RULES

A .hvibe.plus file is a .hvibe file enriched with JavaScript arrow functions inside backtick strings.

- catalog.globals is a backtick field declaring shared constants and variables. Read it first. Declare all its variables once globally. Do not redeclare them anywhere else.
- Every arrow function body found in any backtick field is a reasoning scaffold, not a final implementation. Its JSDoc comment expresses the intent. The code inside is a starting point only.
- You must reason from both the JSDoc and the code, then produce a complete, integrated, and correct implementation. Never copy-paste a sketch as-is. Never treat a sketch as sufficient on its own.
- Sketches may be incomplete, context-unaware, or integration-naive. It is your responsibility to resolve gaps, conflicts, and missing cases using the full context of the file.

GENERATION RULES

1. Generate code immediately upon receiving the file.

2. Read catalog.globals first. Use all declared variables and constants as-is across all features.

3. Resolve all catalog dot-path references before generating.
   RESOLUTION FAILURE PROTOCOL:
   - On failure: retry that reference exactly once.
   - On second failure: stop immediately. Report FAIL and list every unresolved reference. Do not generate any code until the user explicitly validates continuation.
   - On success: report SUCCESS and proceed.

4. Respect stack strictly.

5. Respect depends_on order when sequencing logic.

6. Implement all feature rules as permanent constraints.
   - rules.obligations define required behaviors.
   - rules.constraints define hard limits that must never be violated.

7. Verify all catalog.tests and feature.tests mentally before finalizing. If a test would fail, fix the implementation until it passes.

8. Never skip, omit, or partially implement any feature.

9. If a unit contains a references array, fetch and read each URL or path before generating. If unreachable, state it explicitly. Never simulate or assume reference content.

10. Do not add any features that are not explicitly present in the provided .hvibe files.

11. After generation, evaluate every test defined in catalog.tests and return a report listing each test key followed by PASSED or FAILED.

---

PRIORITY ORDER (highest to lowest)

1. catalog.tests and feature.tests -- must pass, non-negotiable.
2. feature.rules.constraints -- hard limits, never violated.
3. feature.rules.obligations -- required behaviors.
4. feature.description and unit.description -- intent and scope.
5. catalog.logic and blueprints -- reasoning scaffolds, starting points.
6. catalog.globals -- shared state, declared once.

HIGHVIBE SPECIFICATION v0.2.1

.0 FORMAT
A .hvibe.plus file uses JavaScript object notation -- not strict JSON, not JavaScript. Keys are never quoted. Backtick strings are LLM-direct: highest priority, interpret literally. Quoted strings are LLM-indirect: do not ignore, but lower priority.

.1 TOP-LEVEL FIELDS
hvibe_version, app, description (LLM-direct), stack (LLM-direct, respect strictly), catalog (optional, inline or external path), units (each key is a unit name, contains features).

.2 CATALOG
- globals: backtick field declaring shared constants and variables used across all features. Read first.
- logic: arrow functions with JSDoc comments. Treat as reasoning scaffolds.
- code: literal code snippets, use as-is only when explicitly marked as final.
- tests: arrow functions asserting passing conditions. Must pass before finalizing. Keywords: GIVEN, MUST BE, MUST ALWAYS, MUST NEVER, OTHERWISE.

If catalog is a string, treat it as a path to an external file. Multiple catalogs complement each other.

STRICT prefix rule:
If a logic or code entry is prefixed with STRICT:
- It becomes a deterministic algorithm specification.
- Implement it exactly as described. No generalization, abstraction, or architectural extension is allowed.
- Treat it as executable intent, not interpretative guidance.
- Any deviation is considered an implementation error.

.3 UNITS
Each unit has an optional name, an optional description (LLM-direct), an optional references array of external URLs or paths to fetch and read before generating, and a features object.

.4 FEATURE FIELDS
- description (LLM-direct)
- inputs (LLM-direct)
- rules (optional, local -- see .5)
- tests: catalog dot-path refs, must pass before finalizing.
- blueprints: catalog dot-path refs or backtick instructions.
- depends_on: feature and optional when, both LLM-direct.

.5 FEATURE RULES -- local, permanent, implement strictly
- obligations: behaviors to implement.
- constraints: hard limits to enforce.
- Keywords: MUST BE, MUST ALWAYS, MUST NEVER, OTHERWISE.

.6 LOCK FILE -- .hvibe.lock
A companion file recording which features are locked from a previous session.
Format: { hvibe_lock_version, locked_at, features: { "unit.feature": { locked: true | false } } }
- locked: true -- feature is final. Do not touch it.
- locked: false or absent -- feature is pending generation.
- If a locked feature must be modified, flag it and wait for user approval before proceeding.
