LOCK FILE

Before reading the .hvibe file, check if a .hvibe.lock file is present.

If a lock file is present:
- Read it first.
- For every feature marked locked: true, treat its existing implementation as final. Do not regenerate, refactor, or reinterpret it.
- If a locked feature must change to satisfy a new requirement, stop and explicitly flag it to the user before modifying anything. Do not proceed until the user validates the change.
- Focus generation exclusively on features marked locked: false or absent from the lock.

If no lock file is present, proceed normally with full generation.

1. Generate code immediately upon receiving the file.

2. Resolve all catalog dot-path references before generating.
   RESOLUTION FAILURE PROTOCOL:
   - On failure: retry that reference exactly once.
   - On second failure: stop immediately. Report FAIL and list every unresolved reference. Do not generate any code until the user explicitly validates continuation.
   - On success: report SUCCESS and proceed.

3. Respect stack strictly.

4. Respect depends_on order when sequencing logic.

5. Implement all feature rules as permanent constraints.
   - rules.obligations define required behaviors.
   - rules.constraints define hard limits that must never be violated.

6. Verify all catalog.tests and feature.tests mentally before finalizing. If a test would fail, fix the implementation until it passes.

7. Never skip, omit, or partially implement any feature.

8. If a unit contains a references array, fetch and read each URL or path before generating. If unreachable, state it explicitly. Never simulate or assume reference content.

9. Do not add any features that are not explicitly present in the provided .hvibe files.

10. After generation, evaluate every test defined in catalog.tests and return a report listing each test key followed by PASSED or FAILED.

HIGHVIBE SPECIFICATION v0.2.1

.0 FORMAT
A .hvibe file uses JavaScript object notation -- not strict JSON, not JavaScript. Keys are never quoted. Backtick strings are LLM-direct: highest priority, interpret literally. Quoted strings are LLM-indirect: do not ignore, but lower priority.

.1 TOP-LEVEL FIELDS
hvibe_version, app, description (LLM-direct), stack (LLM-direct, respect strictly), catalog (optional, inline or external path), units (each key is a unit name, contains features).

.2 CATALOG -- optional, global, all values LLM-direct
- logic: algorithms or conditions in natural language.
- code: literal code snippets, use as-is.
- tests: assertions to verify before finalizing, adjust until all pass. Keywords: GIVEN, MUST BE, MUST ALWAYS, MUST NEVER, MUST NOT, OTHERWISE.

If catalog is a string, treat it as a path to an external catalog file. Multiple catalogs complement each other.

STRICT prefix rule:
If a logic or code entry is prefixed with STRICT:
- It becomes a deterministic algorithm specification.
- Implement it exactly as described. No generalization, abstraction, or architectural extension is allowed.
- Treat it as executable intent, not interpretative guidance.
- Any deviation is considered an implementation error.

.3 UNITS
Each unit has an optional name, an optional description (LLM-direct), an optional references array of external URLs or paths to fetch and read before generating, and a features object.

.4 FEATURE FIELDS
- description (LLM-direct)
- inputs (LLM-direct)
- rules (optional, local -- see .5)
- tests: catalog dot-path refs, must pass before finalizing.
- blueprints: implementation hints, catalog dot-path refs or backtick instructions.
- depends_on: feature and optional when, both LLM-direct.

.5 FEATURE RULES -- local, permanent, implement strictly
- obligations: behaviors to implement.
- constraints: hard limits to enforce.
- Keywords: MUST BE, MUST ALWAYS, MUST NEVER, OTHERWISE.

.6 LOCK FILE -- .hvibe.lock
A companion file recording which features are locked from a previous session.
Format: { hvibe_lock_version, locked_at, features: { "unit.feature": { locked: true | false } } }
- locked: true -- feature is final. Do not touch it.
- locked: false or absent -- feature is pending generation.
- If a locked feature must be modified, flag it and wait for user approval before proceeding.
